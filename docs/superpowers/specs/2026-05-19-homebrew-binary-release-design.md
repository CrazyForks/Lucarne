# Homebrew Binary Release Design

Date: 2026-05-19
Status: approved for implementation planning

## Problem

Current `Formula/lucarned.rb` installs from the Git repository and runs `cargo install --path crates/lucarned`. Every Homebrew install rebuilds the Rust workspace and pulls Rust/build dependencies. That makes `brew install lucarned` slow and makes the Rust toolchain part of the user install path.

## Goal

Make Homebrew install download a prebuilt `lucarned` binary for macOS Apple Silicon and Intel:

- `aarch64-apple-darwin`
- `x86_64-apple-darwin`

Homebrew users should not need Rust or Cargo for normal installs. Source builds may remain available through `--HEAD` later, but normal formula installs should use release artifacts.

## Non-goals

- Linux Homebrew binaries are not included in this first design.
- Windows packages are not included.
- Homebrew bottle infrastructure is not required for the first release.
- App notarization/signing is not required for the first release because the formula installs a CLI daemon binary, not a `.app` bundle.

## Chosen approach

Use GitHub Actions to build macOS release tarballs, upload them to GitHub Releases, then update the Homebrew formula to select the correct tarball by CPU architecture.

This keeps Homebrew fast and simple:

1. User runs `brew tap tuchg/Lucarne https://github.com/tuchg/Lucarne`.
2. User runs `brew install lucarned`.
3. Formula detects `Hardware::CPU.arm?` or Intel.
4. Formula downloads the matching release tarball and installs `bin/lucarned`.
5. No Rust build happens on the user's machine.

## Release artifacts

For tag `v0.1.0`, CI produces:

```text
lucarned-v0.1.0-aarch64-apple-darwin.tar.gz
lucarned-v0.1.0-x86_64-apple-darwin.tar.gz
checksums.txt
```

Each tarball contains:

```text
bin/lucarned
LICENSE
README.md
```

`checksums.txt` contains SHA-256 checksums for both tarballs. The formula stores the same per-architecture SHA-256 values inline so Homebrew can verify downloads.

## GitHub Actions workflow

Add `.github/workflows/release.yml`.

Triggers:

```yaml
on:
  push:
    tags:
      - "v*"
  workflow_dispatch:
```

Build matrix:

```text
macos-14-arm64 -> aarch64-apple-darwin
macos-15-intel -> x86_64-apple-darwin
```

Build command follows project rules:

```bash
cargo +nightly build --release -Zbuild-dir-new-layout --locked -p lucarned
```

The workflow validates that the tag version matches `workspace.package.version` in `Cargo.toml`. For example, tag `v0.1.0` must match workspace version `0.1.0`.

## Release dependency resolution

Release CI builds the checked-out source with `--locked` so Cargo uses the committed `Cargo.lock` exactly. The repository uses the crates.io dependency range `wechat-ilink = "0"` in `crates/lucarne-wechat/Cargo.toml`, and the resolved crate version lives in `Cargo.lock` like any other dependency.

CI must not run package-specific dependency update steps during release builds. Dependency updates happen before release by updating and committing `Cargo.lock`.

## Formula design

`Formula/lucarned.rb` changes from source build to binary install.

Expected structure:

- Formula metadata stays: `desc`, `homepage`, `version`, `license`.
- macOS ARM branch points to `lucarned-v0.1.0-aarch64-apple-darwin.tar.gz` and stores that tarball's exact SHA-256.
- macOS Intel branch points to `lucarned-v0.1.0-x86_64-apple-darwin.tar.gz` and stores that tarball's exact SHA-256.
- `head "https://github.com/tuchg/Lucarne.git", branch: "main"` remains for explicit HEAD source installs.
- `def install` only runs `bin.install "bin/lucarned"` for release tarballs.

During implementation, the literal SHA-256 values are generated by the release workflow and written into the formula update commit. After the first binary release is published, `main` contains only exact SHA-256 hex values for published assets.

Remove normal-install build dependencies:

```ruby
depends_on "pkg-config" => :build
depends_on "rust" => :build
```

The `head` stanza can remain for future source installs. If `brew install --HEAD lucarned` is supported, HEAD install can keep the old Cargo build path behind `if build.head?`; normal versioned installs use tarballs only.

## Runtime dependencies

Expected macOS release binaries do not need `openssl@3` because `native-tls` uses macOS Security.framework on Apple targets. Implementation must verify each built binary with:

```bash
otool -L target/release/lucarned
```

Formula dependency rule:

- If `otool -L` shows only system libraries/frameworks, remove `depends_on "openssl@3"`.
- If CI discovers a Homebrew dylib dependency, keep the matching runtime `depends_on` and fail release if the dependency is not declared.

## Formula update flow

Release workflow should update formula after assets exist:

1. Build both tarballs.
2. Compute SHA-256 values.
3. Create or update GitHub Release for the tag.
4. Checkout `main`.
5. Replace formula URLs, version, and SHA-256 values.
6. Commit `brew: update lucarned vX.Y.Z` to `main`.

If branch protection blocks direct push, the same workflow creates a pull request with the formula update instead. The project currently uses direct `main` pushes, so direct commit is the default path.

## Data flow

```text
tag vX.Y.Z
  -> release workflow
  -> build locked Cargo source using committed Cargo.lock
  -> build arm64 on macos-14-arm64
  -> build x86_64 on macos-15-intel
  -> package tarballs
  -> upload GitHub Release assets
  -> compute sha256
  -> update Formula/lucarned.rb on main
  -> brew users install binary tarball
```

## Error handling

CI fails before publishing formula updates when:

- Tag version and Cargo workspace version differ.
- Committed `Cargo.lock` is missing, stale, or incompatible with `--locked` build.
- Cargo build fails.
- `lucarned --version` or a smoke command fails.
- Tarball layout does not contain `bin/lucarned`.
- SHA-256 generation fails.
- `otool -L` shows undeclared Homebrew dylib dependencies.
- Formula install test fails on either macOS architecture.

Formula update must not happen unless both architecture tarballs are built, uploaded, and verified.

## Testing strategy

CI release jobs:

1. Build `lucarned` on both macOS architectures.
2. Run `target/release/lucarned` smoke test with `HOME` isolated.
3. Run `otool -L` dependency check.
4. Package tarball and validate layout with `tar -tzf`.
5. Compute SHA-256.
6. After formula update, run Homebrew install test on both architectures:

```bash
brew install --formula ./Formula/lucarned.rb
brew test lucarned
```

Local developer verification before committing workflow/formula changes:

```bash
cargo +nightly check -Zbuild-dir-new-layout -p lucarned
ruby -c Formula/lucarned.rb
```

Full release verification happens in GitHub Actions because both macOS architectures are required.

## Rollout

1. Add release workflow.
2. Change formula to support binary release URLs and optional HEAD source build.
3. Tag `v0.1.0` or next version matching `Cargo.toml`.
4. Let CI build/upload assets and update formula.
5. Verify `brew install lucarned` on Apple Silicon and Intel installs without Rust.
6. Keep remote `main` as source of the Homebrew tap.
